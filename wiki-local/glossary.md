# Trammel glossary

| Term | Definition |
|------|------------|
| **Beam** | One variant in a bounded set of exploration branches. Each beam applies the same steps in a different order using a named strategy (bottom_up, top_down, risk_first, critical_path, cohesion, minimal_change, leaf_first, hub_first, test_adjacent). |
| **Strategy** | JSON-serializable plan: `steps` (ordered work units with dependencies), `dependency_graph`, `constraints`, `goal_fingerprint`. May come from import analysis or from a stored recipe. |
| **Step** | Atomic work unit within a strategy: `{file, symbols, description, rationale, depends_on}`. Steps are ordered by dependency graph and executed/verified individually or in sequence. |
| **Dependency graph** | Map of file â†’ [files it imports from], built by AST analysis of `Import`/`ImportFrom` statements. Used for topological ordering of steps. |
| **Harness** | `ExecutionHarness`: temp copy of the project, optional file writes from edits, subprocess test run. Supports full run, single-step verification, and incremental verification. |
| **Edit** | Dict consumed by the harness: `path` or `file` plus `content` to write; may also carry `step_index` and `task` for planner metadata. Without `content`, no file is changed. |
| **Recipe** | Successful strategy associated with a goal pattern; stored by SHA-256 of canonical JSON (`sig`). Tracks `successes` and `failures` counts for quality signal. |
| **Pattern** | Truncated goal string stored next to a recipe for retrieval similarity (not the full strategy). |
| **Plan** | Row in `plans`: goal, strategy snapshot, status (`pending`/`running`/`completed`/`failed`), step progress tracking. |
| **Constraint** | Persistent failure record: type (`dependency`/`incompatible`/`requires`/`avoid`), description, context. Active constraints are loaded during decomposition to prevent repeating known-bad approaches. |
| **Trajectory** | One harness run for a beam: plan_id, beam_id, strategy_variant, steps_completed, outcome, failure_reason. |
| **Failure analysis** | Structured extraction from test output: error_type, message, file, line, suggestion. Used to create constraints. |
| **Trigram bag cosine** | Similarity between two strings built from overlapping 3-character windows; counts are aligned on the union of trigram types before cosine. Used as one component of `goal_similarity`. Minimum threshold 0.3. |
| **Incremental verification** | Per-step harness runs where each step is verified in isolation (with prior edits applied). Stops at first failure with structured analysis. |
| **Bottom-up** | Beam strategy: modify dependencies first, then dependents. Safest ordering. |
| **Top-down** | Beam strategy: modify API surface and entry points first, then internals. |
| **Risk-first** | Beam strategy: modify most-imported (highest coupling) files first. |
| **Critical-path** | Beam strategy: longest dependency chain first. Iterative stack-based DFS depth computation (with `in_stack` cycle detection) identifies bottleneck files; feedback prioritizes them. Converted from recursive to iterative to fix stack overflow on deep graphs (e.g., Guava's 1.66M-edge Java import graph). |
| **Cohesion** | Beam strategy: flood-fill connected components in the dependency graph, process tightly coupled groups contiguously (largest component first), topological sort within each component. |
| **Minimal-change** | Beam strategy: fewest symbols first. Quick wins that catch trivial failures early. |
| **Goal normalization** | Preprocessing step for recipe matching: lowercase + abbreviation expansion via `_ABBREVIATIONS` (~40 common coding abbreviations) + verb synonym replacement via `_VERB_SYNONYMS` (40+ verb variants mapped to 9 canonical forms). Applied before trigram indexing and similarity scoring. |
| **Goal similarity** | Blended similarity function: 0.3 trigram cosine + 0.4 word Jaccard + 0.3 word substring on normalized text. Used by `retrieve_best_recipe` for scoring and replaces raw `trigram_bag_cosine` for recipe matching. |
| **Word Jaccard** | Word-level Jaccard similarity: `|intersection| / |union|` on the word sets of two strings. Used as a component (0.4 weight) in `goal_similarity`. |
| **Verb synonyms** | `_VERB_SYNONYMS` dict in `text_similarity.py` (re-exported from `utils.py` for back-compat) mapping 40+ verb variants to 9 canonical forms (e.g., refactor/rewrite/reorganize all map to "restructure"). Enables recipe matching across paraphrased goals. |
| **Path aliases** | TypeScript `compilerOptions.paths` + `baseUrl` from `tsconfig.json`. Read by `_read_ts_path_aliases(root)` and resolved by `_resolve_alias()` in `TypeScriptAnalyzer` for non-relative alias imports. |
| **Inverted trigram index** | The `recipe_trigrams` table mapping individual trigrams to recipe signatures. Enables sub-linear recipe retrieval by narrowing candidates before exact cosine computation. |
| **Transaction** | Explicit `BEGIN IMMEDIATE` / `COMMIT` / `ROLLBACK` block with exponential backoff retry on `SQLITE_BUSY`. Ensures atomicity for multi-statement mutations and agent isolation. |
| **Constraint enforcement** | `_apply_constraints` in `core.py` that acts on active constraints: `avoid` skips steps, `dependency` injects ordering, `incompatible` marks conflict metadata, `requires` adds prerequisite steps. |
| **trammel.db** | Default SQLite path for recipes, recipe_trigrams, recipe_files, plans, steps, constraints, and trajectories â€” 7 tables (override with `db_path`/`--db`). |
| **Strategy registry** | Pluggable registry in `strategies.py` for beam strategies. `register_strategy(name, description, fn)` adds new strategies; `get_strategies()` returns all registered `StrategyEntry` items. Nine built-in strategies auto-registered at module load (bottom_up, top_down, risk_first, critical_path, cohesion, minimal_change, leaf_first, hub_first, test_adjacent). Strategy functions use unified signature `(steps, dep_graph) -> steps`. |
| **Strategy learning** | When `explore_trajectories` receives an optional `store`, it calls `get_strategy_stats()` to aggregate trajectory outcomes by strategy variant, then sorts strategies by historical success rate. Strategies that have succeeded more often are tried first. |
| **Language analyzer** | Implementation of the `LanguageAnalyzer` protocol in `analyzers.py`/`analyzers_ext.py`/`analyzers_ext2.py`. Provides `collect_symbols`, `analyze_imports`, `test_command`, and `error_patterns` for a specific language. Built-in: `PythonAnalyzer` (AST-based), `TypeScriptAnalyzer` (regex-based, stdlib-only), `GoAnalyzer` (regex-based, reads `go.mod`), `RustAnalyzer` (regex-based, `use crate::`/`mod`), `CppAnalyzer` (regex-based, `#include "..."`), `JavaAnalyzer` (regex-based, package imports), `CSharpAnalyzer` (regex-based, `using`), `RubyAnalyzer` (regex-based, `require`/`require_relative`), `PhpAnalyzer` (regex-based, `use`/`require`/`include`), `SwiftAnalyzer` (regex-based, `import`), `DartAnalyzer` (regex-based, `import`/`part`), `ZigAnalyzer` (regex-based, `@import`). `detect_language()` auto-selects by config file or extension prevalence. Registry supports 15 languages. |
| **Composite scoring** | Recipe retrieval scoring when `context_files` are provided. Weighted combination with recency: text similarity (0.4) + file overlap via Jaccard index (0.25) + success ratio (0.15) + recency (0.2, 30-day half-life). Without `context_files`, falls back to text-only scoring for backward compatibility. |
| **Recipe files** | The `recipe_files` table mapping file paths to recipe signatures. Populated on `save_recipe` from strategy step file paths. Enables structural matching via Jaccard overlap during recipe retrieval. Auto-backfilled via `_backfill_files()` for existing databases. |
| **System prompt** | `SYSTEM_PROMPT.md` â€” a reference orchestration guide for LLM clients describing the plan-verify-store loop. Shows how an LLM should use Trammel's MCP tools to decompose goals, verify steps, store recipes, and propagate constraints. |
| **Rebuild trigram index** | `_rebuild_trigram_index` in `store.py` (renamed from `_backfill_trigrams`). Rebuilds all trigrams with normalized text on init, ensuring the inverted index reflects current normalization rules. |
| **GoAnalyzer** | Regex-based `LanguageAnalyzer` for Go projects. Reads `go.mod` for the module path. Resolves internal imports (imports matching the module path) to project-relative file paths. Uses shared `_collect_symbols_regex` helper for symbol collection. |
| **RustAnalyzer** | Regex-based `LanguageAnalyzer` for Rust projects. Resolves `use crate::` imports and `mod` declarations to project-relative file paths. Uses shared `_collect_symbols_regex` helper for symbol collection. |
| **Word substring score** | `word_substring_score(a, b)` in `utils.py`. Partial word matching function that checks for substring overlap between the word sets of two strings. Rewards partial matches that pure Jaccard misses. Used as a component (0.3 weight) in `goal_similarity`. |
| **Recency weighting** | Time-based decay factor in composite recipe scoring. Uses a 30-day half-life: recently updated recipes score higher than stale ones. Weight 0.2 in composite scoring (text 0.4, files 0.25, success 0.15, recency 0.2). |
| **Comment stripping** | `_strip_c_comments` in `utils.py`. Removes single-line (`//`) and multi-line (`/* */`) comments from source code before symbol and import detection, preventing false matches in commented-out code. All analyzers now use shared comment strippers from `utils.py`. |
| **CppAnalyzer** | Regex-based `LanguageAnalyzer` for C/C++ projects in `analyzers_ext.py`. Handles `.c/.cpp/.cc/.cxx/.h/.hpp/.hxx`. 5-pattern symbol detection: template functions, qualified functions (static/inline/constexpr), operator overloading, constructor/destructor, macro-prefixed functions (EXPORT_API etc). `#include "..."` import resolution with comment stripping. Registered as "cpp" and "c". |
| **JavaAnalyzer** | Regex-based `LanguageAnalyzer` for Java/Kotlin projects in `analyzers_ext.py`. Handles `.java/.kt/.kts`. Symbol detection for class, interface, enum, fun, object, and @interface declarations. `_detect_source_roots(project_root)` reads `build.gradle`/`build.gradle.kts`/`pom.xml` for standard source directories; falls back to project root. `analyze_imports` walks detected source roots. Registered as "java" and "kotlin". |
| **Recipe pruning** | `RecipeStore.prune_recipes(max_age_days=90, min_success_ratio=0.1)` in `store.py`. Removes stale, low-quality recipes and cascades deletes to `recipe_trigrams` and `recipe_files`. |
| **Parallel beam execution** | `plan_and_execute` in `__init__.py` runs beams concurrently via `concurrent.futures.ProcessPoolExecutor` (stdlib). `_run_beam` wraps a single beam harness run; `_run_beams_parallel` dispatches all beams to the pool. Falls back to sequential on systems where process spawning fails. |
| **Base copy caching** | `ExecutionHarness.prepare_base(project_root)` creates one filtered base copy of the project; `run_from_base(edits, base_dir)` copies from it. Avoids re-filtering ignored directories per beam. |
| **Dry-run mode** | `--dry-run` CLI flag in `cli.py`. Runs `explore()` instead of `plan_and_execute()`, producing a strategy without verification or harness execution. |
| **analyzers_ext** | Extension module (`trammel/analyzers_ext.py`, ~400 LOC) holding `GoAnalyzer`, `RustAnalyzer`, `CppAnalyzer`, `JavaAnalyzer`, and the shared `_collect_symbols_regex` helper. Split from `analyzers.py` to stay under 500 LOC per file. Re-exported from `analyzers.py` for backward compatibility. |
| **analyzers_ext2** | Extension module (`trammel/analyzers_ext2.py`, ~480 LOC) holding `CSharpAnalyzer`, `RubyAnalyzer`, `PhpAnalyzer`, `SwiftAnalyzer`, `DartAnalyzer`, `ZigAnalyzer`. Second extension split to stay under 500 LOC per file. Re-exported from `analyzers.py` for backward compatibility. |
| **CSharpAnalyzer** | Regex-based `LanguageAnalyzer` for C# projects in `analyzers_ext2.py`. Handles `.cs` files. Symbol detection for class, interface, struct, enum, record, and delegate declarations. `using` import resolution. Registered as "csharp". |
| **RubyAnalyzer** | Regex-based `LanguageAnalyzer` for Ruby projects in `analyzers_ext2.py`. Handles `.rb` files. Symbol detection for class, module, and def declarations. `require`/`require_relative` import resolution. Registered as "ruby". |
| **PhpAnalyzer** | Regex-based `LanguageAnalyzer` for PHP projects in `analyzers_ext2.py`. Handles `.php` files. Symbol detection for class, interface, trait, enum, and function declarations. `use`/`require`/`include` import resolution. Registered as "php". |
| **SwiftAnalyzer** | Regex-based `LanguageAnalyzer` for Swift projects in `analyzers_ext2.py`. Handles `.swift` files. Symbol detection for class, struct, enum, protocol, func, and actor declarations. `import` resolution. Registered as "swift". |
| **DartAnalyzer** | Regex-based `LanguageAnalyzer` for Dart projects in `analyzers_ext2.py`. Handles `.dart` files. Symbol detection for class, mixin, extension, enum, and typedef declarations. `import`/`part` resolution. Registered as "dart". |
| **ZigAnalyzer** | Regex-based `LanguageAnalyzer` for Zig projects in `analyzers_ext2.py`. Handles `.zig` files. Symbol detection for pub fn, const, struct, enum, and union declarations. `@import` resolution. Registered as "zig". |
| **MCP server** | Trammel exposed as 31 tools via Model Context Protocol stdio transport (`prune_plans` added in 3.12.1). Tool JSON schemas live in `tool_schemas.py` (new in 3.12.0); dispatch lives in `mcp_server.py`. Language enums include 15 entries. Works standalone; cooperates with Stele and Chisel when co-installed. |
| **store_recipes** | Module (`trammel/store_recipes.py`, ~373 LOC after 3.12.0, down from 1223) holding `RecipeStoreMixin` with recipe CRUD. Retrieval methods live in `store_retrieval.py` (`RecipeRetrievalMixin`); scaffold recipes in `store_scaffolds.py`; structural fingerprinting helpers in `recipe_fingerprints.py`. Plan/step management lives in `store_plans.py` (`PlanStoreMixin`); telemetry in `store_telemetry.py`. `RecipeStore` composes all six mixins. Split in 3.12.0 to keep every store module under 500 LOC. |
| **Plan resumption** | `get_plan_progress(plan_id)` in `store.py`. Returns plan state with accumulated `prior_edits` from all passed steps, `remaining_steps`, and `next_step_index`. Enables LLM clients to resume a failed plan from the last successful step instead of re-planning from scratch. Exposed as `resume` MCP tool. |
| **Recipe validation** | `validate_recipes(project_root)` in `store_recipes.py`. Checks `recipe_files` entries against the current project filesystem. Removes stale file entries for files that no longer exist. Cascade-prunes recipes whose files are entirely missing. Exposed as `validate_recipes` MCP tool. |
| **Config-file detection** | `_detect_from_config(project_root)` in `analyzers.py`. Detects project language from config files before falling back to extension counting. Checks: Cargo.toml (rust), go.mod (go), tsconfig.json/package.json (typescript), build.gradle/pom.xml (java), CMakeLists.txt (cpp), pyproject.toml/setup.py (python), Package.swift (swift), build.zig (zig), pubspec.yaml (dart), .csproj/.sln (csharp), Gemfile (ruby), composer.json (php). More reliable than extension counting for projects with generated files or mixed codebases. |
| **Shared file collection** | `_collect_project_files(root, extensions)` in `utils.py`. Shared helper that walks a project directory, skips ignored dirs, and collects files matching given extensions. Replaces duplicated `os.walk` patterns in `TypeScriptAnalyzer._collect_files`, `CppAnalyzer._collect_files`, and `RustAnalyzer.analyze_imports`. |
| **Shared namespace walker** | `_walk_and_map_namespaces(project_root, extensions, namespace_re, preprocess, source_roots=None)` in `utils.py`. Walks project files, reads sources with comment stripping, extracts namespaces/packages via regex, and returns `(namespace_to_files, file_to_source)` dicts. Used by CSharp, PHP, and Java analyzers. Supports optional `source_roots` for Java-style multi-root projects. |
| **Scope** | Optional subdirectory parameter on `decompose`, `explore`, `plan_and_execute`, CLI (`--scope`), and MCP tools. When provided, analysis is scoped to `os.path.join(project_root, scope)` while the full project root remains available for test execution. Enables monorepo workflows where a 10k+ file repository only needs a subset analyzed. |
| **Concurrent write safety** | Trammel's SQLite store uses `BEGIN IMMEDIATE` transactions with exponential backoff retry on `SQLITE_BUSY`. Validated with threading tests: 4 concurrent threads performing 5 operations each on plans, recipes, and constraints all complete without errors. |
| **Abbreviation expansion** | `_ABBREVIATIONS` dict in `utils.py` mapping ~40 common coding abbreviations to their full forms (gcâ†’garbage collector, dbâ†’database, authâ†’authentication, apiâ†’application programming interface, etc.). Applied by `normalize_goal` before verb synonym replacement. Enables recipe matching across abbreviated goals (e.g., "optimize GC" matches "optimize garbage collector" at 0.86+ similarity). |
| **Analysis metadata** | `analysis_meta` dict returned by `decompose` containing `language`, `scope`, `files_analyzed`, `dep_files`, `dep_edges`, `timing_s` (with `symbols`, `imports`, `total` sub-keys), and optional `warning` when the detected language analyzer may not be native (e.g., C# falling back to C++). Helps LLMs understand analysis cost and coverage. |
| **Estimate tool** | `estimate` MCP tool for quick file count without running full analysis. Returns `language`, `matching_files`, and `recommendation` ("use scope" if >5000 files, "full analysis OK" otherwise). Helps LLMs decide whether to scope before analyzing large repositories. |
| **Implicit dependency** | A dependency not expressed in import statements but inferred from naming conventions, shared state access patterns, or learned co-occurrence. Trammel estimates ~50% of actual dependencies are implicit. |
| **Naming convention inference** | Dependency inference via naming patterns: `XService` â†’ `XModel`, `XRoute` â†’ `XService`, `XController` â†’ `XService` + `XModel`. Suffix-based base-name extraction handles CamelCase, snake_case, and kebab-case. |
| **Shared state coupling** | Implicit dependency detected when two modules read/write the same data file. `SharedStateDetector` tracks `_file_access_map` and `_module_access_map` to surface modules that implicitly depend on each other via file I/O rather than imports. |
| **Pattern-based inference** | Dependency inference via learned co-occurrence from existing import graphs. `PatternLearner.learn_from_import_graph()` builds `_direct_deps`, `_dep_pair_cooccurrence`, and `_suffix_patterns` to predict what a new module likely depends on based on observed patterns. |
| **Hybrid dependency graph** | Dependency graph combining explicit imports (AST-analyzed) with implicit dependencies (naming + shared state + learned patterns). Produced by `ImplicitDependencyGraphEngine.get_hybrid_dependency_graph()`. |
| **Trammel blind spots** | Gaps in dependency coverage where Trammel's import analysis misses ~50% of actual dependencies. Identified by `ImplicitDependencyGraphEngine.get_gap_analysis()`. Reported as `trammelBlindSpots` with `count`, `percent`, and `category_breakdown`. |
| **ImplicitDependencyGraphEngine** | Engine (`trammel/implicit_deps.py`, ~268 LOC after 3.12.0 split) coordinating four components: `NamingConventionEngine` + `SharedStateDetector` (both in `implicit_deps_engines.py`), `PatternLearner` (in `pattern_learner.py`), and the top-level engine in `implicit_deps.py`. Naming-convention rules are externalized to `trammel/data/patterns.json` and loaded at import via `pattern_config.py`. |
| **Pattern configuration** | `trammel/data/patterns.json` holds `naming_convention_rules` (Serviceâ†’Model/Route etc.), `infrastructure_patterns`, `file_role_patterns`, `goal_role_patterns`, and `default_convention_confidence`. `pattern_config.py` loads, validates, and caches the file. New in 3.12.0 â€” previously these were inline Python tables in `implicit_deps.py` and `store_recipes.py`. |
| **text_similarity module** | `trammel/text_similarity.py` â€” trigram similarity + goal normalization APIs extracted from `utils.py` in 3.12.0. Hosts `_trigram_list`, `trigram_signature`, `unique_trigrams`, `trigram_bag_cosine`, `_cosine`, `_VERB_SYNONYMS`, `_ABBREVIATIONS`, `_TECH_SYNONYMS`, `expand_goal_terms`, `normalize_goal`, `word_jaccard`, `word_substring_score`, `goal_similarity`. All re-exported from `utils.py` for back-compat. |
| **scaffold_validation module** | `trammel/scaffold_validation.py` â€” scaffold DAG metrics (`compute_scaffold_dag_metrics`) and pre-flight validation (`validate_scaffold`: cycles, duplicates, self-reference, over-constrained, missing deps) extracted from `utils.py` in 3.12.0. Re-exported from `utils.py`. |
| **planner_helpers module** | `trammel/planner_helpers.py` â€” extracted free functions used by `Planner` (new in 3.12.0): `decompose_scaffold_only(planner, ...)`, `explore_trajectories(planner, ...)`, `suggest_strategy(store, goal, language)`. `Planner._decompose_scaffold_only`, `Planner.explore_trajectories`, and the legacy `_suggest_strategy` name delegate to these. |
| **scaffold_creation module** | `trammel/scaffold_creation.py` â€” goalâ†’scaffold heuristic pipeline extracted from `scaffold_logic.py` in 3.12.0: `_infer_file_name`, `_sibling_convention_clones`, `_fallback_directories`, `_directories_for_role_hints`, `_detect_layered_architecture`, `_creation_hints`, `_generate_creation_steps` plus constants (`_ROLE_DIR_SUBSTRINGS`, `_FALLBACK_ROLE_DIRS`, `_LAYER_DIRS`, `_LAYER_PATTERNS`). Re-exported from `scaffold_logic.py`. |
| **language_detection module** | `trammel/language_detection.py` â€” `detect_language`, `_detect_from_config`, `_detect_from_trammel_config`, `_LANG_EXTENSIONS` extracted from `analyzers.py` in 3.12.0. Re-exported from `analyzers.py`. |
| **analyzer_resolvers module** | `trammel/analyzer_resolvers.py` â€” per-language import-graph resolvers extracted from `analyzer_engine.py` in 3.12.0: `_resolve_go_imports`, `_resolve_rust_imports` (with `_resolve_rust_mod`, `_read_cargo_crates`), `_resolve_cpp_imports`, `_resolve_java_imports` + `_detect_java_source_roots`, `_resolve_csharp_imports`, `_resolve_ruby_imports`, `_resolve_php_imports`, `_resolve_swift_imports` + `_build_swift_module_map`, `_resolve_dart_imports`, `_resolve_zig_imports`. | **Plan pruning** | `RecipeStore.prune_plans(max_age_days=7, status="pending")` in `store_plans.py` (new in 3.12.1). | **Transaction-nesting fix (3.12.1)** | Python 3.14's default `sqlite3.isolation_level=""` auto-began an implicit transaction before DML, so any unbracketed write left an open implicit transaction that broke the next `BEGIN IMMEDIATE` with *"cannot start a transaction within a transaction"*. Fixed in `db_connect` (`isolation_level=None` â€” manual mode), `transaction()` (SAVEPOINT-aware on nested entry, explicit COMMIT/ROLLBACK), and `recipe_index.backfill_recipe_index` (now wrapped in `transaction()`). | **Scope-aware scaffold matching (3.12.1)** | `_scaffold_matches_scope(scaffold, scope)` in `core.py` rejects scaffold-recipe matches whose files all live outside the queried `scope`. | **Corrupt strategy_json hardening (3.12.1)** | `_safe_json` in `store_plans.py` wraps `json.loads` for the `strategy`, `scaffold`, and step JSON columns. Corrupt rows return synthetic empty payloads plus a `_corrupted_fields: [...]` marker; `get_plan`, `resume`, and `available_steps` all degrade gracefully instead of raising `JSONDecodeError`. |
| **recipe_files dedup (3.12.1)** | Older `save_recipe` paths could append the same `(recipe_sig, file_path)` row many times (320+ duplicates seen on `c54a62b9a7e6`). Fix is three-layered: `_dedupe_recipe_files()` collapses duplicates on init; a unique index `idx_recipe_files_unique` prevents new ones; `_insert_file_entries` switched to `INSERT OR IGNORE`. `list_recipes` also de-dupes the file list it returns as a final safety net. |
| **get_recipe / near_match parity (3.12.1)** | When `_handle_get_recipe` finds nothing above the 0.55 auto-match threshold, it now falls back to `retrieve_near_matches` and returns those â€” same retrieval path as `explore.near_match_recipes`, so the two tools can no longer disagree on whether a goal is matchable. |
| **Suffix-based inference** | Dependency inference by extracting base name and suffix from filenames. `_extract_base_name` and `_extract_suffix` in `implicit_deps.py` handle CamelCase, snake_case, kebab-case, and plain names. |
| **Infrastructure inference** | Implicit dependency from files matching `INFRASTRUCTURE_PATTERNS` (e.g., `config`, `store`, `db`, `cache`). Any module matching infrastructure names is treated as implicitly depended upon by modules with the same base name. |
| **Three inference strategies** | The `ImplicitDependencyGraphEngine` combines three complementary strategies: (1) naming conventions (Serviceâ†’Model/Route), (2) shared state coupling (file I/O), (3) learned co-occurrence (observed import patterns). Confidence scores range 0.5â€“0.95. |
